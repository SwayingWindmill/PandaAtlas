# PandaAtlas 前端设计原则与规范优先级

- **状态：** 已确认，作为后续前端信息架构、系统边界、原型与质量评审的项目政策
- **确认日期：** 2026-07-15
- **Wayfinder 票：** [Define PandaAtlas frontend design principles and standards precedence](https://github.com/SwayingWindmill/PandaAtlas/issues/33)
- **父地图：** [Wayfinder: PandaAtlas 前端系统性改进](https://github.com/SwayingWindmill/PandaAtlas/issues/31)
- **依据：** [PandaAtlas 公开 Beta 产品需求文档](../product/panda-atlas-public-beta-prd.md) 与 [前端系统及证据缺口审计](../research/frontend-system-audit-2026-07-15.md)

## 1. 决策摘要

PandaAtlas 采用 **“保留可信内核，系统性重构公开体验”** 的改造姿态。

公开前端不是营销站、通用 Dashboard 或沉浸式数字展览，而是一个面向熊猫爱好者与普通公众的、可查证且可继续深入探索的公共档案。其品牌与产品共同呈现 **“温暖的自然史档案馆”**：事实清晰、来源透明、语气有温度，但不把拟人化、视觉戏剧性或内容完整感置于真实性之前。

后续决策必须遵守以下优先级：

1. 产品事实、领域可信度、许可、隐私与安全；
2. 无障碍与核心任务可完成性；
3. 本文定义的 PandaAtlas 项目设计原则与系统规范；
4. 仍与前三层一致的现有项目约定；
5. Impeccable 中被本文采纳的系统化规则；
6. taste-skill 中被本文采纳的品牌与编辑场景辅助规则。

发生冲突时，不取平均折中，由更高优先级直接覆盖更低优先级。

## 2. 规范性语言

本文使用以下强度：

- **必须 / 不得：** 发布或合并前必须满足，不允许仅以审美、开发便利或历史实现为由豁免。
- **应该 / 不应该：** 默认执行；偏离时必须在对应设计或实施记录中说明用户收益、风险与验证方式。
- **可以：** 在不违反更高层约束时可选择使用。

## 3. 首要用户与任务

### 3.1 用户优先级

公开前端的用户优先级为：

1. **熊猫爱好者与普通公众**；
2. **科普、教育与研究型访客**；
3. 资料编辑者、审核者和管理员仅在共享基础系统涉及公开层时纳入本次前端政策。

辅助技术用户、移动端用户与低速网络用户不是独立的次级人群，而是所有核心任务的覆盖条件。不得通过另做一个功能缩水的“无障碍版”“移动简版”或“文字备用页”来满足要求。

### 3.2 核心任务顺序

公开体验首先帮助用户：

1. 通过名称、别名、旧称、链接或关联内容找到一只熊猫；
2. 快速确认身份、出生信息、当前场所、已知亲本、档案等级与最后核实时间；
3. 沿家族、迁移足迹、机构与相关个体继续探索；
4. 在需要时查看来源、冲突、不确定性、版本与修订；
5. 对公开信息执行收藏、分享、纠错或数据下载等次级任务。

首页、导航和检索不得优先展示宏观指标、品牌口号、复杂地图或视觉故事，而让“找到并理解一只熊猫”退居次位。

## 4. 主要物理使用场景

PandaAtlas 的基准使用场景是 **“移动端快速查证，桌面端深入探索”**。

### 4.1 移动端快速查证

典型用户从搜索引擎、社交分享、聊天链接或站内搜索进入一只熊猫档案，在几分钟内完成：

- 确认是否为目标个体；
- 查看关键事实与当前场所；
- 打开父母、子女或迁移记录；
- 必要时核验来源；
- 分享当前页面或保存该个体。

关键事实、导航与来源入口必须在普通手机和触控环境中成立，不得要求横屏、精确悬停或大型画布。

### 4.2 桌面端深入探索

桌面场景用于：

- 比较多条档案；
- 浏览更完整的时间线、家族关系和迁移记录；
- 操作地图与谱系，同时使用结构化列表；
- 查看来源、版本、许可和修订；
- 下载公开数据。

桌面端可以提高信息密度，但不得把移动端需要的关键事实、来源或等价路径变成桌面专属能力。

### 4.3 加载与恢复

- 首屏不得立即加载完整地图、复杂谱系或非必要大型媒体。
- 影响任务含义的搜索、筛选、排序、地图模式、选中对象、谱系焦点和章节定位必须写入 URL。
- 返回、刷新、分享和无 JavaScript 场景必须尽可能保留任务语义。

## 5. 改造姿态：保留什么，重构什么

### 5.1 必须保留的可信内核

以下能力属于产品资产，不因视觉重构而丢失：

- 公开结论与来源的关联；
- 最后核实时间，以及事件发生日期、来源发布日期、录入日期之间的区分；
- 已核实、待核实、冲突、过期、受限等真实状态；
- 档案版本、Public Schema 版本与修订摘要；
- 媒体许可、无图和仅来源链接状态；
- 中英文规范 URL、语言元数据与核心旅程等价性；
- 地图与谱系的结构化文本等价路径；
- JavaScript 降级可读性；
- 已建立的构建、浏览器、无障碍和 Release Gate 证据纪律。

### 5.2 允许系统性重构的公开体验

以下内容不因“已经存在”而自动获得保留权：

- 首页信息结构与视觉表达；
- 全局导航与移动端导航；
- 图鉴检索和浏览工作区；
- 旧 `/atlas/[slug]` 详情契约；
- 地图与谱系的交互布局；
- 卡片、徽章、圆角、阴影和间距习惯；
- `globals.css` 中的页面级风格所有权；
- 静默 fallback、生成式故事、性格、标签、排序分数和无来源展示元数据；
- 不能完成真实任务的 CTA、指标和社交证明。

### 5.3 不采用的改造方式

- 不只做表层换肤而保留冲突的数据与路由契约。
- 不从零重写可信数据模型、发布工作流或全部技术栈。
- 不因某个外部规范推荐而批量更换组件库或增加依赖。
- 不在跨页面原则、旅程与系统边界未确定前进行大规模视觉翻新。

## 6. 品牌与产品 register

PandaAtlas 使用 **一个共享系统、两个受控 register**，不是每个页面各自形成主题。

### 6.1 共享基础

品牌页与产品页必须共享：

- 可信度与内容真实性规则；
- 色彩、字体、间距、形状、焦点、状态与动效 Token；
- 全局导航与语言行为；
- 无障碍、响应式、性能和媒体许可要求；
- “温暖的自然史档案馆”语气。

### 6.2 品牌与编辑 register

适用于首页、产品介绍、经过审核的故事和编辑专题。

可以使用：

- 更有节奏的留白；
- 少量衬线标题；
- 更强的图片、地图细节或档案材料构图；
- 有明确事实依据的编辑叙事。

不得使用：

- 无来源数字、奖项、用户量或保护成效；
- 内部设计过程、实现框架或“这一版”的说明作为公开文案；
- 纯装饰性沉浸动画压过产品任务；
- 模板化 SaaS Hero、空泛使命宣言或无实际去向的 CTA。

### 6.3 产品与档案 register

适用于检索、个体档案、机构、地图、谱系、来源与数据下载。

必须表现为：

- 任务优先；
- 信息层级稳定；
- 中等偏高密度；
- 无衬线主导；
- 状态、来源和时间明确；
- 图形与结构化文本并列；
- 完整交互状态可验证。

不得为了与首页“看起来完全一致”而牺牲数据密度、结构化表达或操作效率。

## 7. 自然历史档案馆的视觉与文案性格

### 7.1 应呈现的性格

- **温暖：** 对生命个体保持尊重和好奇，不把页面写成行政数据库。
- **克制：** 不夸大事实，不用装饰制造虚假的完整度或权威感。
- **可查证：** 事实与来源、时间、状态之间的关系易于理解。
- **有材料感：** 可以借鉴自然史出版物、档案卡、田野记录、地图标注和馆藏目录的组织方式。
- **当代：** 不复制仿古纸张、邮票、图章或手账装饰来模拟“档案感”。

### 7.2 必须避免的性格

- 儿童化、吉祥物化或过度可爱化；
- 把熊猫拟人化描述当作事实；
- 冷峻、行政化、表单堆叠式数据库观感；
- 奢华杂志、时尚大片或纯品牌展览观感；
- 卡通竹子、熊猫黑白、国潮纹样等符号的无差别堆叠；
- 通过徽章、认证图标或权威口号替代真实来源与修订记录。

## 8. 色彩策略

项目采用 **“自然低饱和 + 档案中性色 + 语义状态色”**。

### 8.1 基础色

- 低饱和苔藓绿、竹叶绿、土壤棕等自然色用于品牌强调与有限导航线索。
- 深墨灰、暖灰、象牙白、纸张米白等中性色承担正文、背景、分隔与层级。
- 熊猫黑白仅作为品牌识别和局部高对比资源，不作为整站唯一色彩逻辑。

### 8.2 状态色

已核实、待核实、存在冲突、历史或过期、受限、成功、警告和错误必须使用稳定的语义 Token。状态不得从页面局部的装饰色临时推导。

每个状态必须同时具备文字、图标、形状或结构提示，不得只靠颜色区分。

### 8.3 禁止项

- 高饱和荧光色作为主视觉；
- 大片纯黑制造“高级感”；
- 绿色渐变作为默认强调方式；
- 每个功能独立创造强调色；
- 未验证对比度的浅灰正文、透明文字和图片叠字。

所有正文、控件、焦点、图例和状态组合以 WCAG 2.2 AA 对比度为最低门槛；最终达标必须以自动与人工证据共同确认。

## 9. 字体与排版策略

项目采用 **“无衬线承担产品阅读，衬线仅用于少量档案标题”**。

### 9.1 字体职责

- 正文、表格、定义列表、筛选、按钮、来源、日期、地图和谱系标签必须使用高可读无衬线字体。
- 页面标题、章节标题或编辑引语可以克制使用衬线字体，形成出版物与档案目录感。
- 衬线字体不得用于密集交互、长段小字号正文或需要快速扫描的事实字段。

### 9.2 中英文关系

- 中文和英文分别选择适合其文字系统的字体，不要求字形完全同构。
- 两种语言的字号、字重、行高、信息层级与控件尺寸必须功能等价。
- 不得通过缩小英文或中文来强行保持相同换行与卡片高度。

### 9.3 加载与性能

- 默认使用高质量系统字体栈，或少量自托管字体和必要字重。
- 不默认依赖第三方远程字体服务。
- 新字体必须证明中文覆盖、数字与日期可读性、许可、加载策略和性能成本。
- 不得加载多套相近字体或完整字重集合只为装饰差异。

### 9.4 排版禁止项

- 大量全大写英文；
- 过度字距；
- 极细字重；
- 装饰性手写字体；
- 超大标题挤压首屏任务；
- 无限制行宽与密集长段；
- 为视觉整齐截断事实、来源标题或状态说明。

## 10. 动效策略

项目采用 **低干扰、解释性动效**。

### 10.1 允许的作用

动效可以用于：

- 解释筛选结果更新；
- 表明展开、收起、选择、收藏或保存结果；
- 帮助理解谱系节点或地图对象之间的空间关系；
- 维持界面连续性，避免状态突变造成迷失。

### 10.2 约束

- 动效必须有信息作用，不得只用于“让页面活起来”。
- 不得阻塞操作、隐藏内容或要求用户等待动画结束。
- `prefers-reduced-motion` 必须覆盖 CSS 动画以及 MapLibre、谱系画布、轮播等命令式动画。
- 减少动画时，地图定位和谱系跳转应直接完成或使用近乎即时的状态变化。
- 自动播放内容必须可停止；默认不使用自动播放背景视频、持续漂浮、视差或滚动劫持。

## 11. 信息密度与构图

产品界面采用 **中等偏高信息密度**。

- 档案首屏应优先容纳身份、关键事实、当前场所、亲本、档案等级、来源线索和主要操作。
- 桌面端通过网格、定义列表、表格、时间轴和并排结构提高扫描效率。
- 移动端通过顺序重排、分组与渐进展开控制认知负担，不删除关键事实。
- 单页应有一个主要视觉焦点，不通过多个同权大标题、指标块或卡片争夺注意力。
- 数据密集内容优先使用列表、定义列表、时间轴和表格，不把每一项都包装成卡片。

必须避免：

- 卡片套卡片；
- 所有容器都使用大圆角和阴影；
- 徽章、胶囊标签和 tracked eyebrow 泛滥；
- 用大量空白制造“高级感”却拉长任务路径；
- 把结构关系拆成无法比较的独立卡片。

## 12. 无障碍与可信度发布门槛

以下要求不可因视觉效果、品牌一致性、排期或开发便利而豁免。

### 12.1 核心任务等价

核心任务必须在以下条件下可完成：

- 键盘；
- 中文与英文真实读屏环境；
- 200% 放大与文本重排；
- 触控；
- 减少动画；
- 普通手机；
- 低速网络；
- 地图或复杂图形不可用；
- JavaScript 不可用或部分失败。

### 12.2 地图与谱系

- 地图必须始终提供可操作、按语义组织的结构化列表，不得只在地图加载失败时出现。
- 谱系必须始终提供有顺序、有关系类型的结构化表示，不得要求用户理解二维布局。
- 图形视图与结构化视图必须访问同一语义对象，并能相互保持焦点或选择状态。

### 12.3 内容真实性

- 不允许将本地生成内容、fixture、静默 fallback、推测、推荐算法或编辑包装伪装成已发布档案事实。
- 所有公开事实必须根据领域模型区分已核实、待核实、存在冲突、历史记录或最后一次核实记录。
- 没有合规媒体时必须展示明确无图状态，不得复用其他熊猫图片并重新标注。
- 未经支持的故事、性格、偏好、体重、位置、排名、指标和社会证明不得发布。
- 第三方媒体、地图底图和嵌入内容必须有可执行的来源、许可和署名策略。

### 12.4 合规声明

未完成人工键盘、真实读屏、对比度、地图等价、谱系等价和核心流程验收时，不得宣称达到 WCAG 2.2 AA。自动化通过只代表相应自动检查通过。

## 13. 规范优先级与冲突处理

### 13.1 固定优先级

#### 第一层：产品事实与领域边界

包括 PRD、已发布数据语义、来源与修订、媒体许可、隐私、安全、敏感位置和公共投影边界。

任何规范不得要求：

- 隐藏不确定性或冲突；
- 伪造完整度；
- 用装饰性内容补齐空档；
- 暴露受限数据；
- 省略必要许可与署名。

#### 第二层：无障碍与核心任务可完成性

包括 WCAG 2.2 AA 目标、键盘与读屏、移动端、触控、低速网络、非视觉等价路径、JavaScript 降级和减少动画。

视觉布局、品牌表达或技术简化不得破坏核心任务等价性。

#### 第三层：PandaAtlas 项目设计原则

即本文定义的用户、场景、register、色彩、字体、动效、密度和可信度规则。它们是后续设计与评审的直接依据。

#### 第四层：现有项目约定

现有路由、组件、CSS、配色和交互仅在符合前三层时保留。历史实现不是规范来源，也不能以“已有测试依赖”为由保留错误产品契约；测试应随已确认契约迁移。

#### 第五层：Impeccable

仅本文明确采纳的条目进入项目政策。其作用是提供系统化审计、层级、响应式、状态、性能、动效、无障碍和反模板化方法，不覆盖上层产品事实。

#### 第六层：taste-skill

仅用于品牌、首页与编辑内容的 brief inference、anti-slop 和 pre-flight。它不主导档案、检索、表格、地图、谱系或复杂工作流。

### 13.2 冲突处理步骤

遇到规则冲突时，评审者必须依次回答：

1. 该方案是否改变、隐藏或伪造产品事实、来源、许可、隐私或安全边界？若是，否决。
2. 该方案是否使任一核心任务在键盘、读屏、移动端、低速网络、非视觉或降级场景中不可完成？若是，否决。
3. 该方案是否违反本文已确认的 PandaAtlas 原则？若是，除非更新本决策，否则否决。
4. 现有项目约定是否仍服务前三层？若否，迁移旧约定而不是降低新方案。
5. Impeccable 或 taste-skill 的建议是否适用于当前 register？若不适用，不采纳。
6. 若多个低层方案均合规，再以任务效率、维护成本、性能与视觉质量选择。

## 14. Impeccable 的项目政策子集

以下内容成为 PandaAtlas 强制基线：

1. **上下文与审计优先：** 修改前必须理解产品目标、真实内容、现有系统和 register。
2. **产品与品牌区分：** 品牌页可以更表达性，产品页必须任务优先；二者共享系统而非各自漂移。
3. **信息层级优先：** 先确定用户要看到和完成什么，再处理装饰与视觉风格。
4. **完整状态设计：** 交互必须覆盖加载、空、错误、禁用、成功、部分、截断、离线、过期与降级等适用状态。
5. **响应式不是缩放：** 移动端需要重新排序与重组，而不是压缩桌面画布。
6. **无障碍进入设计本体：** 键盘、焦点、语义、对比度、读屏、减少动画与触控必须在设计阶段处理。
7. **性能进入设计本体：** 字体、图片、地图、谱系和客户端数据量必须有加载策略与预算。
8. **动效有职责：** 只使用解释状态和空间关系的动效。
9. **反模板化：** 避免默认 Hero、泛滥卡片、无意义大字、过度圆角、徽章堆叠和装饰性渐变。
10. **生产状态验证：** 评审必须包含真实内容、双语压力、移动端、键盘、错误与降级状态，不只看理想静态稿。

以下内容不自动成为项目要求：

- 任何具体视觉配方、示例配色、字体或构图；
- 与产品真实任务无关的 landing-page 惯例；
- 为满足规范示例而增加的依赖或组件库；
- 与 PandaAtlas 可信度、无障碍或性能冲突的审美规则。

## 15. taste-skill 的项目政策子集

以下内容只在首页、品牌介绍和编辑故事中作为辅助规则：

1. 从真实 brief、内容和品牌性格推导视觉方向；
2. 在重设计前识别现有模板感与视觉陈词滥调；
3. 页面应有明确视觉主张，而不是通用 SaaS、作品集或 AI 生成页面观感；
4. 在交付前执行 anti-slop pre-flight；
5. 视觉表达不能以牺牲可读性、内容真实性和任务效率为代价。

以下做法不适用于产品型表面，不得机械套用：

- 以展示型构图替代结构化信息；
- 通过超大标题、极端留白或实验导航制造视觉差异；
- 把地图、谱系、筛选或表格按 landing page 方式重构；
- 为“独特”而降低可预测性、可扫描性或状态完整性。

## 16. 后续设计与评审的判定清单

任何 IA、系统架构、原型或页面方案进入下一阶段前，至少必须回答：

- 它首先服务哪一个核心用户任务？
- 首屏是否让用户更快确认个体与关键事实？
- 它展示的每项数据、文案、指标和媒体来自哪里？
- 不确定、冲突、过期、fallback 和无内容状态是否诚实？
- 中文、英文、移动端、桌面端、键盘、读屏和无 JavaScript 路径是否等价？
- 地图或谱系不可见时，任务是否仍可完成？
- 状态是否可以分享、刷新和恢复？
- 它属于品牌 register 还是产品 register？是否错误套用了另一种 register？
- 是否使用项目 Token 和共享模式，而不是创造局部视觉系统？
- 动效是否有信息职责，并完整响应减少动画？
- 字体、媒体、地图和客户端数据是否有性能与许可策略？
- 是否出现卡片套卡片、徽章泛滥、无意义大字、装饰性渐变或模板化 Hero？

任何一个涉及产品事实、核心任务等价性或不可豁免发布门槛的问题未回答，方案不得以“后续再补”进入生产实现。

## 17. 对后续 Wayfinder 票的约束

### Map the core public journeys and information architecture

必须围绕“找到一只熊猫 → 确认关键事实 → 沿家族、足迹、机构与来源继续探索”建立主旅程；品牌页与产品页可以使用不同 register，但必须共享导航、语言、可信语义和状态恢复。

### Design the frontend system boundary and token/component architecture

必须建立共享基础 Token 与受控 register，清除每个功能独立定义颜色、半径、阴影、状态和 copy 的模式。系统边界必须区分发布数据、编辑内容、fallback 来源与 view model。

### Define content, data-honesty, media, and UI-state rules

必须把本文的可信度门槛转换为可执行的数据来源、fallback、媒体清单、署名和状态文案规则，不得通过视觉设计掩盖内容缺口。

### Prototype the cross-surface PandaAtlas design language

原型必须同时展示品牌与产品 register、移动端快速查证、桌面端深入探索、双语压力、非视觉地图/谱系路径、fallback/过期状态、键盘焦点与减少动画，不得只展示理想桌面首页。

### Define frontend quality gates and visual verification

质量门禁必须验证本文所有不可妥协约束，并把自动化证据与人工验收明确分开。视觉质量评审不能替代内容真实性、无障碍或性能证据。

## 18. 非目标

本文不决定：

- 最终路由与导航树；
- 具体 Token 名称和值；
- 具体字体家族、组件 API 或地图底图供应商；
- 页面高保真构图；
- 后端领域模型与 Public API 重设计；
- 实施顺序和提交切片。

这些问题由后续 Wayfinder 票分别解决，但不得违反本文政策。

## 19. 最终决议

PandaAtlas 的前端质量不以“是否像一个漂亮的熊猫网站”为主要判断，而以以下组合判断：

> 用户能否在一个温暖但克制的自然史档案环境中，快速找到并理解一只熊猫，继续探索其家族与足迹，并始终知道信息来自哪里、何时核实、有哪些不确定性，以及在任何核心访问条件下如何完成同一任务。

因此，现有可信档案语义与证据纪律继续作为内核；公开信息架构、视觉语言、组件和交互可以系统性重构。任何外部设计规范只在服务这一目标时有效。
